    扩展类加载器加载jre/lib/ext中的类System.getProperty("java.ext.dirs");

双亲委派机制

除了引导类加载器，其他的每一个类加载器都有一个超类加载器，当加载器接到加载任务的时候，并不会立即进行加载，而是将此重任交给其超类加载器，每一层加载器皆是如此，直至委派至至高无上的引导类加载器，如果超类加载器无法加载该类时就任务退回至下一层，每一层皆是如此。

protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // First, check if the class has already been loaded
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            long t0 = System.nanoTime();
            try {
                // 如果超类加载器不是引导类加载器就由超类加载，否则引导类试图加载
                /* 所以如果我们自定义加载器将其父类加载器设置为null，那么，
                就不会有App/Ext加载器试图去加载目标类，只会有引导类加载器和自定义的加载器加载目标类

                如果将自定义加载器的父类加载器设置为Ext，就不会有app加载器试图加载目标类
                */
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // ClassNotFoundException thrown if class not found
                // from the non-null parent class loader
            }
            // 此处真正进行加载，所以自定义加载器继承ClassLoader重写findClass方法。
            if (c == null) {
                // If still not found, then invoke findClass in order
                // to find the class.
                long t1 = System.nanoTime();
                c = findClass(name);

                // this is the defining class loader; record the stats
                sun.misc.PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                sun.misc.PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                sun.misc.PerfCounter.getFindClasses().increment();
            }
        }
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}

  • 为何要使用双亲委派机制？

有效保证一个类的全局唯一性和安全性吧，比如我自定义一个java.lang.StringBuilder类，通过自定义加载器进行加载，任务委派至引导类加载器，发现该类全路径在引导类加载器的加载路径中，引导类加载器就会加载到jdk的那个StringBuilder类，返回至我自定义的加载器就不会进行加载自定义的StringBuilder了。

自定义类加载器

public class MyClassLoader extends ClassLoader {
    
    private final static Path DEFAULT_CLASS_DIR = Paths.get("E:");

    private final Path classDir;

    public MyClassLoader() {
        super();
        this.classDir = DEFAULT_CLASS_DIR;
    }

    public MyClassLoader(String classDir) {
        super();
        this.classDir = Paths.get(classDir);
    }

    public MyClassLoader(String classDir, ClassLoader parent) {
        super(parent);
        this.classDir = Paths.get(classDir);
    }

    @Override
    protected Class<?> findClass(String name)
            throws ClassNotFoundException {
        // 读取目标class文件
        byte[] classBytes = this.readclassBytes(name);
        if (classBytes.length == 0)
            throw new ClassNotFoundException("Can not load the class" + name);
        return this.defineClass(name, classBytes, 0, classBytes.length);
    }

    private byte[] readclassBytes(String name)
            throws ClassNotFoundException {
        String classPath = name.replace(".", "/");
        Path classFullPath = classDir.resolve(Paths.get(classPath + ".class"));
        if (!classFullPath.toFile().exists())
            throw new ClassNotFoundException("The class" + name + "not found");
        try (ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            Files.copy(classFullPath, baos);
            return baos.toByteArray();
        } catch (Exception e) {
            throw new ClassNotFoundException("load the class" + name + " occur error.", e);
        }
    }

    @Override
    public String toString() {
        return "MyClassLoader";
    }
}

破坏双亲委派机制

因双亲委派的逻辑是loadClass方法，所以继承ClassLoader重写其loadClass方法即可

@Override
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        Class<?> aClass = findLoadedClass(name);
        if (aClass == null) {
            /* 双亲委派的原则是试图由super classloader加载，无法加载再返回加载任务
                此处是直接由自己加载，自己不能加载再交给super classloader加载
            */
            /* 点解要排除java/javax开头的呢？ 
                因为findClass =》defineClass =》preDefineClass
                if ((name != null) && name.startsWith("java.")) {
                    throw new SecurityException
                        ("Prohibited package name: " +
                        name.substring(0, name.lastIndexOf('.')));
                }
            */
            if (name.startsWith("java."))) {
                try {
                    // app classloader，当然其load原则是双亲委派机制的
                    aClass = getSystemClassLoader().loadClass(name);
                } catch (Exception e) {

                }
            } else {
                try {
                    aClass = this.findClass(name);
                } catch (Exception e) {

                }
                if (aClass == null) {
                    if (getParent() != null) {
                        aClass = getParent().loadClass(name);
                    } else {
                        aClass = getSystemClassLoader().loadClass(name);
                    }
                }
            }
        }
        if (aClass == null) {
            throw new ClassNotFoundException("fucking...");
        }
        if (resolve) {
            resolveClass(aClass);
        }
        return aClass;
    }
}

类加载器命名空间

每一个类加载器实例都有各自的命名空间，命名空间由其所有父类加载器和该加载器构成，因此每一个class在每个加载器中仅有独一无二的class实例，但是在JVM中存在多个class实例是可能的。

MyClassLoader loader1 = new MyClassLoader();
Class<?> aClass = loader1.loadClass("com.tab.test.C");

MyClassLoader loader2 = new MyClassLoader();
Class<?> bClass = loader2.loadClass("com.tab.test.C");
// aClass/bClass虽然是由同一个加载器加载，但是是由不同的加载器实例加载，所以aClass != bClass
// 那如果是不同的加载器，那结果自然也会不同
Class<?> cClass = loader2.loadClass("com.tab.test.C");
// bClass == cClass

加载过程

加载

1.通过一个类的全限定名来获取定义此类的二进制字节流
此字节流的获取方式可为：
从ZIP读取、网络中获取、运行时计算生成（动态代理，ProxyGenerator.generateProxyClass）、
其他文件生成（JSP)、数据库中读取等等等
2.将该字节流所代表的静态存储结构转化为方法区的运行时数据结构
3.在Java堆中生成一个代表这个类的class对象，作为方法区这些数据的访问入口。

连接

验证

确保class文件字节流是健康且符合当前虚拟机的要求，其过程大致如下：

1. 文件格式验证：魔术，版本号，常量池常量类型（tag标志）等等
2. 元数据验证：语义分析，是否有父类，继承是否正确（比如父类是否被final修饰），
非抽象类是否实现其父类抽象类接口的方法等等
3. 字节码验证：
4. 符号引用验证：发生在连接的第三个阶段-解析，将符号引用转为直接引用的时候

准备

为类变量在方法区/元空间分配内存并设置零值，实例变量是在对象实例化时随对象一起在堆中分配内存

static int i =1; 准备阶段是赋值为0，而赋值为1是在初始化阶段执行<clinit>()时的putstatic指令。当然如果时静态常量则不会将赋值放入<clinit>()中，在javac编译时就会为其生成ConstantValue属性，在准备阶段就其直接赋值为ConstantValue属性值

解析

将符号引用（引用目标不一定在内存中）转为直接引用（引用目标必定已经在内存之中）

解析动作主要针对以下四类符号引用进行：

1.类或接口
2.字段
3.类方法
4.接口方法

初始化

根据程序员通过程序制定的主观计划去初始化类变量和其他资源，即执行<clinit>()方法

<clinit>()是由编译器自动收集类中的所有类变量的赋值动作和静态语句块（非静态语句块会被编译到每一个构造方法之中的首行，若无静态语句块和对静态变量的赋值操作，则无clinit方法）中的语句合并产生的，且顺序由源文件中顺序决定，so，静态代码块中不能访问在后面定义的变量

<clinit>()不需要显示调用父类构造器，虚拟机会保证子类的clinit方法执行之前，父类的clinit方法已经执行完毕，并且虚拟机会保证<clinit>()在多线程环境中被正确地加锁和同步

初始化时机

虚拟机规范严格规定有且只有以下四种情况必须立即对类进行“初始化"(当然加载，验证，准备自然是在此之前）：

1.遇到new,getstatic,putstatic,invokestatic这四条字节码指令时，如果类尚未进行过初始化，则需
先进行初始化，其场景是：new实例化对象，读取或设置类的静态字段（final static自然不可修改，为何读取不会呢：已在编译期把结果放
入常量池，编译时已完成宏替换），调用类的静态方法。
2.使用java.lanng.reflect包的方法对类进行反射调用时
3.当初始化一个类时，如果父类还没有初始化，则需先触发其父类初始化
4当虚拟机启动时，需指定一个要执行的主类（包含main()方法的类），虚拟机会先初始化这个类

这四种场景中的行为称为对一个类进行主动引用，除此之外所有引用类的方式，都不会触发初始化，称为被动引用。eg:

public class C {
    public static void main(String[] args) {
        System.out.println(Sub.i); //super's clinit 2
    }
}
class Super {
    static {
        System.out.print("super's clinit ");
    }
    static int i = 2;
}
class Sub extends Super {
    static {
        System.out.print("sub's clinit ");
    }
    static final String NAME = "007";
}

常量在编译阶段会存入调用类的常量池中，本质上没有直接引用到定义常量的类，因此不会触发定义常量的类的初始化，在编译阶段，被调用类的常量会存储到调用类的常量池，所以编译之后调用类和被调用类就没有半毛钱关系了。

接口的初始化与类的初始化大致相同，接口中不能有static代码块，但编译器仍会为接口生成<clinit>()类构造器，用于初始化接口中所定义的成员变量，（接口只能定义静态常量，静态常量直接便以为ConstantValue,又何来clinit?）接口和类的区别在于"有且仅有的第三种“，初始化类要求初始化其所有父类，but初始化接口时，并不要求初始化其父接口，只有在使用到父接口（比如父接口中定义的常量? 常量被编译为ConstantValue,不是不会触发类的初始化吗？）时才会初始化。

    static F f = new F();

    static int b = 1;

    public F() {
        a = 11;
        b = a;
        System.out.print(a + " ");

    }

    public static void main(String[] args) {
        System.out.print(a + " ");
        System.out.print(b + " ");
        // 11 11 11 1
    }
}
