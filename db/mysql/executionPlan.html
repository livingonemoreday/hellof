explain

查看执行计划 explain + sql 

* id: 表的读取顺序（从大到小，从上至下）
* select_type: 读表类型（SIMPLE-简单查询，PRIMARY-最外层的查询，SUBQUERY-select或where后的子查
               询，DERIVED-from后的衍生查询，UNION-union后的查询，UNION RESULT-a union b的结果）
* table: 表名
    。 若select_type为UNION RESULT，table则为<union1,2>（此处1，2指被union的查询id）
    。 若从select_type为DERIVED的结果中获取，table则为<derived9>（此处9指DERIVED的查询id)
* partitions: 分区
* type: 数据访问类型-衡量查询性能指标，从优至劣如下
    。 SYSTEM: 表中有且仅有一条数据，不管是否存在索引
    。 CONST: 通过主键或唯一索引一次扫描即可定位，where条件是常量值
    。 EQ_REF: 唯一性索引扫描，出现于主键或唯一索引，where条件非常量值，是其他关联条件等
    。 REF: 非唯一性索引扫描，eg:普通索引
    。 RANG: 索引范围扫描，between and > < in 等（普通索引字段范围查找未走索引， 不知何故）
    。 INDEX: 全索引表扫描
    。 ALL: 全数据表扫描
* possible_keys: 可能用到的索引
* key: 实际用到的索引
* key_len: 索引字节数
* ref: 关联字段，若为常量值则显示CONST
* rows: 检索到目标记录所需读取行数，越小越佳
* filtered: 存储引擎返回的数据，满足过滤条件的百分比
* extra: 额外信息
    。 USING FILESORT: 文件排序，无法利用索引排序，恶劣
    。 USING TEMPORARY: 临时文件，常见于order by，group by，恶劣
    。 USING INDEX: 覆盖索引，直接从索引中获取数据，优秀

索引

注意事项

* 复合索引按索引顺序使用，跳跃索引列、范围查找后的索引列失效
* 索引列上不要进行函数、计算等操作
* 通配符不要放左边（若非要如此，则使用覆盖索引）
